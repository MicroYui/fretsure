# Oracle 0.2 — 公共输入、判决与统计信任门

> **状态（2026-07-16）**：**DONE（软件信任门已关闭）**。在独立闭门的 MusicXML-first 提交上重放后的快照为 `1092 passed, 6 deselected`、本地代理全量 `1098 passed`、`1098 collected`；ruff、mypy(strict)、`uv lock --check`、`git diff --check`、wheel/sdist 与两类 clean-venv smoke 的最终证据见下方。本文冻结从内部、已归一化原型升级到可供未来 Web/API/MCP 使用的公共边界合同；真人经验有效性与主流 notation application 的正兼容证据仍是明确的独立 open gate。

**Goal:** 将 playability checker 与 Python package 同步升级到 `0.2.0`，让任意不可信 Tab、MusicIR、solver target、profile、tier、运行配置和真人标签数据在进入几何/时序谓词、生成工作或统计汇总前被确定性合同接住。

**Architecture:** JSON 层只负责严格、无 coercion 的结构解析；`oracle.input` 统一 Tab/profile/instrument/solver 控制边界，MusicIR、tier、Trace、benchmark 与 gold/statistics 各有独立 typed barrier，并在验证后只消费 detached snapshot。输入有效性与 playability 判决正交：非法输入抛 typed error，绝不伪装成 RED/AMBER/GREEN。每个有效判决绑定 checker version、profile version、profile canonical SHA-256 与 input schema version。

## 1. 冻结合同

- package：`fretsure-oracle==0.2.0`。
- checker：`oracle@0.2.0`。
- 输入 schema：`tab-input@0.2.0`。
- profile：已有 `small@0.1` / `median@0.1` / `large@0.1` 名称不改；新增绑定全部模型参数的 canonical SHA-256。
- faithfulness：继续为独立的 `fidelity@0.2.0`，不随本次 playability 升级改义。
- MusicXML importer：闭门时继续为 `musicxml@0.1.0`；本计划不扩大输入乐谱语义。后继安全 `.mxl` reader 是独立能力增量，须单独版本化，不能回写本闭门快照。

输入失败语义：

- Python oracle 抛 `OracleInputError`；solver 抛 `SolverInputError`；MusicIR 与 benchmark 控制分别抛 `IRInputError`、`BenchmarkInputError`。
- 未来 HTTP/MCP adapter 映射为 `INVALID_INPUT`，不制造 checker verdict。
- 空 Tab 无可认证命题，因此非法；空 solver target 是合法请求但无解，返回 typed `Infeasible(EMPTY_TARGET, onset=None, ...)`。
- 有效 Tab 上同一物理弦的半开 sounding interval 重叠是模型违反，判 RED；前音恰在新音 onset 结束合法。
- profile-relative instrument range（例如物理域内的 fret 23 超过某 profile 的 fret 22）仍是 RED，不是输入非法。

统计失败语义：

- zero-GREEN 的 false-accept rate / Clopper–Pearson upper bound 是 `no_green` / `None`，不是 `0.0`。
- 无 certified observations 或退化 marginals 的 Cohen κ 是 explicit `undefined` / `None`。
- 空 item 集的 pass^k 是 `no_items` / `None`；分母必须保留。
- 非法 human-label row 在汇总前带 row/line typed fail；不能返回部分 confusion matrix。

## 2. 资源与类型 envelope

- ordinary guitar Tab：精确六弦、严格递增 MIDI tuning；精确 non-negative capo；非空；字段使用 exact `Fraction` / exact int，不接受 bool、float/string coercion。
- public physical fret envelope：`0..36`；profile 的 `max_fret` 也在该范围内。
- MIDI sounding pitch：`0..127`。
- Tab/solver target：最多 20,000 notes；单 onset 最多 64 attacks/targets。
- Fraction numerator/denominator：各最多 256 bits。
- tempo：finite `1..1000` BPM；beats/bar：exact int `1..32`；solver beam：exact int `1..1024`。
- MusicIR：exact tuple 中最多 20,000 notes + 20,000 chords，累计文本最多 10 Mi characters；所有公共 pipeline/agent/arranger/critic/伴奏入口先深快照；Fraction 组件最多 256 bits，source tempo 为 finite `1..1000`，拍号分子 `1..32`、分母 `1..64`。
- untrusted Tab JSON：严格字段集、拒 duplicate key / NaN / Infinity / unknown field / non-canonical fraction，并有 payload、note-count、fraction-token 与 bit-length 上限。
- Python dataclass/list 边界在 validation 前做有界深快照，checker、solver 与 reliability 只消费该快照；低层 mutation 不能在“验证后、使用前”改写判决、搜索规模或统计分母。
- Tab serializer 在构建 note dictionaries / 调用 JSON encoder 前精确证明 canonical 输出不超过 10 MiB；Trace 只接受有界 plain JSON + canonical Fraction，拒绝 NaN、任意对象 hook、循环/过深/过大数据，并在 `json.dumps` 前精确计算含 JSON escaping 的 compact UTF-8 字节数，随后与 encoder 结果交叉核对 10 MiB 上限。
- solver 只接受 inert built-in list/tuple；不执行任意 duck-typed Sequence 的 `__len__` / `__getitem__`。
- 直接 `arrange`/`repair`/`simplify` Python 入口与 pipeline 共用 64 candidates / 64 repair iterations 上限，且控制值不做 bool/truthiness coercion。
- tier 控制先做 detached snapshot 与 exact-domain validation；横按 overlap 诊断使用按六弦反向扫描的 `O(6n)` 实现，同时保持既有诊断顺序/重数。
- benchmark 在生成 corpus 或调用 LLM factory 前拒绝非法控制：seed 为 signed 63-bit exact int，items `1..1000`，bars `1..64`，`items*bars <= 4096`，paired 为 exact bool。
- gold 文件/内存输入：最多 100,000 rows、64 MiB、200,000 notes、2,000,000 checker-work units、1,000,000 physical lines；单行 JSON depth 64 / 250,000 nodes，全体最多 1,000,000 nodes。加载后以深快照、digest 与来源记录防 mutation/TOCTOU 冒充原始行。
- solver 的 frame configs、fingerings/config、beam、final full-oracle checks 分别至多 48、64、1024、16；总门为 12,000,000 个加权 work units。上界显式覆盖 raw Cartesian 枚举、config retention/diversity、beam sort/rescan、backpointer 重建，以及三套 profile 下 checker 的排序、attack-pair 与 active-sounding-pair 工作。
- proposal-only 入口复用同一结构/乐器/profile/tempo 域校验，但不为并未发生的假想 beam search 计费；真正进入 solver/repair/simplify 时仍以实际 beam 重新过完整 work gate。

## 5. 真人边界

本计划不需要真人，不因真人 gold 尚未收集而阻塞。它只证明公共软件边界、模型内 checker 和统计管线不会把非法/无证据状态说成有效结论。

仍需真人才能关闭：现实世界 GREEN false-accept rate、profile/tier 到真实琴手的映射、AMBER 经验带宽、难度与 musicality 校准，以及“真实琴手一定能弹”的强主张。在此之前，对外只说：GREEN 通过 `oracle@0.2.0` 在明确 profile/参数下的简化几何 + 有限时序/速率模型。
